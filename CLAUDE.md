## 프로젝트 목적

- 대상: NestJS 초보자
- 방식: 17개 챕터를 순서대로 학습하면서 **블로그 API 서버**를 점진적으로 완성
- 언어: 한국어 (모든 문서, 설명, 주석)
- 검증: 모든 챕터 내용은 **NestJS 공식 문서(https://docs.nestjs.com/)** 를 기반으로 작성되어야 하며, 공식 문서에서 확인되지 않은 내용은 포함하지 않는다

## 문서 작성 규칙

> **상세 규칙**: `.claude/chapter-rules.md` 파일에 검증 체크리스트, 헤더 방식별 챕터 목록, 코드 블록 규칙 등이 상세히 정리되어 있다. 챕터 작성 시 반드시 함께 참조한다.

### 챕터 구성 (반드시 준수)

모든 챕터 문서(`01-*.md` ~ `17-*.md`)는 다음 **4단계 구조**를 따릅니다. **네 단계 모두 반드시 존재해야 하고 이 순서를 지켜야 합니다.**

```
1단계: 개념 학습      — 무엇인지, 왜 필요한지, 다이어그램/비유 포함
2단계: 기본 예제      — 독립적인 작은 예제로 개념 확인 (직접 실행 가능)
3단계: 블로그 API 적용 — 배운 내용을 블로그 프로젝트에 즉시 적용
4단계: 정리          — ## 정리 (요약 표) + ## 다음 챕터 예고 (챕터 17 제외)
```

### 섹션 헤더 방식 (방식 A 단일 표준)

모든 챕터는 명시적 단계 헤더를 사용한다:

```
# 1단계: 개념 학습
# 2단계: 기본 예제
# 3단계: 블로그 API 적용
```

- 단계 헤더는 반드시 **H1(`#`)** 을 사용한다. `##`로 쓰면 일반 섹션과 구분이 안 돼 구조가 무너진다.
- 단계 내부의 세부 섹션은 `## N. 섹션명` 형태로 번호를 붙여 구분할 수 있다.
- TOC의 아이템 번호와 본문 섹션 번호는 반드시 일치해야 한다. 다른 챕터에서 복붙한 후 번호를 수정하지 않으면 불일치가 발생한다.

### 챕터 제목 형식

```markdown
# 챕터 N - 주제명
```

- 반드시 `챕터 N - 주제` 형식 (한글, 숫자, 하이픈)
- 예: `# 챕터 1 - Module`, `# 챕터 12 - Authentication (인증)`

### 이전/다음 챕터 연결

**이전 챕터 요약** — 제목 바로 아래 블록쿼트:
```markdown
> **이전 챕터 요약**: 챕터 N-1에서 ~를 만들었다. 이번 챕터에서는 ~를 학습한다.
```

**다음 챕터 예고** — 파일 맨 끝 전용 섹션:
```markdown
## 다음 챕터 예고

챕터 N+1에서는 **주제**를 학습한다. 블로그 API에 ~기능이 추가된다.
```

- 챕터 1: 이전 챕터 요약 없음 (첫 챕터)
- 챕터 17: 다음 챕터 예고 없음 (마지막 챕터)

### 블록쿼트 스타일 (3종으로 통일)

```markdown
> **팁:** 추가 도움이 되는 정보
> **주의:** 실수하기 쉬운 부분
> **참고:** 배경 지식이나 보조 설명
```

**절대 금지 패턴** (콜론 이중 사용 오류):
- ❌ `> **팁:**: ` — 볼드 "팁:" 뒤에 콜론이 하나 더 붙는 오류
- ✅ `> **팁:** ` — 올바른 형식

### 코드 블록

- 언어: `typescript`
- 파일명을 **첫 줄 주석**으로 표시:
  ```typescript
  // src/app.module.ts
  ```

### 챕터 마무리 구조

모든 챕터 파일은 다음 순서로 마무리한다:

```markdown
## 정리

| 개념 | 설명 |
|------|------|
| ... | ... |

## 다음 챕터 예고

챕터 N+1에서는 ...
```

- `## 정리` 가 반드시 먼저, `## 다음 챕터 예고` 가 파일 맨 끝
- 챕터 1: 이전 챕터 요약 없음 / 챕터 17: `## 다음 챕터 예고` 없음
- **`## 정리` 이후에는 본문 내용을 절대 추가하지 않는다.** 심화 섹션이나 보충 설명은 반드시 `## 정리` 앞, 3단계 내부에 배치한다.

### 데코레이터 인라인 링크

챕터 본문에서 데코레이터가 언급될 때 `references/decorators.md`로 인라인 링크 연결:
- 코드 블록 **밖**에서만 링크 적용
- 형식: `` [`@Controller`](references/decorators.md#앵커) ``
- 앵커는 decorators.md의 실제 섹션 헤더 기준 (GitHub 마크다운 앵커 규칙 적용)
- **주의**: 챕터 파일은 `docs/` 안에 있고 references도 `docs/references/`에 있으므로 `../references/`가 아닌 `references/`로 연결해야 한다

다음 상황이 발생하면 **반드시 CLAUDE.md도 함께 갱신**한다:

- 새 챕터(18~) 또는 새 레퍼런스 파일 추가
- 문서 작성 규칙이 새로 생기거나 기존 규칙이 변경될 때
- 챕터 구성 방식, 링크 형식, 헤더 형식 등 구조적 규칙이 바뀔 때

CLAUDE.md가 실제 상태와 달라지면 에이전트가 잘못된 기준으로 작업할 수 있다.

### 블로그 API 예제 코드 품질

3단계(블로그 API 적용)의 예제 코드는 이미 만들어진 서비스와 유틸리티를 재사용해야 한다:

- **하드코딩 금지**: 날짜 포맷, 슬러그 생성 등 공통 로직은 인라인으로 중복 작성하지 않는다
- **`CommonService` 우선 사용**: `formatDate()`, `generateSlug()` 등 이미 정의된 메서드를 활용한다
- **이 규칙은 3단계에만 적용**: 2단계(기본 예제)는 개념 설명이 목적이므로 간결한 인라인 코드를 허용한다

```typescript
// ❌ 3단계에서 금지
createdAt: new Date().toISOString().split('T')[0],

// ✅ 3단계에서 권장
createdAt: this.commonService.formatDate(new Date()),
```

### 공식 문서 기반 원칙

모든 챕터 내용은 **https://docs.nestjs.com/** 을 기준으로 한다:

- API, 데코레이터, 설정 옵션 등은 공식 문서에서 직접 확인 후 작성한다
- 공식 문서에 없는 서드파티 패턴이나 커뮤니티 관행은 명확히 구분하거나 포함하지 않는다
- NestJS 버전에 따라 동작이 달라질 수 있는 내용은 주의사항으로 표기한다
- 챕터 작성 또는 수정 시 해당 주제의 공식 문서 페이지를 먼저 확인한다

## references/decorators.md 관리

- 이 파일은 **독립 문서**입니다. 다른 챕터에 의존하지 않고 혼자 읽어도 이해되어야 합니다.
