# 챕터 작성 규칙 (Chapter Writing Rules)

Claude Code가 새 챕터를 작성하거나 기존 챕터를 수정할 때 반드시 준수해야 하는 규칙입니다.

---

## 0. 공식 문서 기반 원칙 (최우선)

**모든 내용은 https://docs.nestjs.com/ 을 기준으로 작성한다.**

- 챕터를 작성하거나 수정하기 전에 해당 주제의 공식 문서 페이지를 반드시 확인한다
- 데코레이터 시그니처, 옵션, 동작 방식은 공식 문서에서 직접 확인한 내용만 기술한다
- 공식 문서에 없는 서드파티 패턴이나 커뮤니티 관행은 포함하지 않거나 명확히 구분 표기한다
- NestJS 버전에 따라 동작이 달라질 수 있는 내용은 `> **주의:**` 블록쿼트로 표기한다

---

## 1. 4단계 흐름 (필수)

모든 챕터는 반드시 다음 **4단계 순서**를 따라야 한다. **네 단계가 모두 존재해야 하고 이 순서를 지켜야 한다**.

```
[1단계] 개념 학습
  ├── 무엇인지 (정의, 비유)
  ├── 왜 필요한지 (동기, 문제 상황)
  ├── 다이어그램 또는 흐름도
  └── 핵심 개념 설명

[2단계] 기본 예제
  ├── 블로그 API와 무관한 독립적인 예제
  ├── 개념을 직접 실행해볼 수 있는 작은 코드
  └── 각 개념별 예제 (1:1 대응)

[3단계] 블로그 API 적용
  ├── 앞에서 배운 개념을 블로그 프로젝트에 적용
  ├── 이전 챕터 코드를 이어받아 기능 추가
  └── curl 또는 실행 결과로 동작 검증

[4단계] 정리
  ├── ## 정리 — 핵심 개념 요약 표
  └── ## 다음 챕터 예고 — 다음 챕터 내용 소개 (챕터 17 제외)
```

### 단계별 검증 체크리스트

새 챕터 완성 전 아래 항목을 모두 확인한다:

- [ ] **공식 문서** — 이 챕터의 내용이 https://docs.nestjs.com/ 에서 확인된 내용인가?

- [ ] **1단계** — "무엇인가"와 "왜 필요한가"가 모두 설명되어 있는가?
- [ ] **1단계** — 비유나 다이어그램(ASCII art, 표)이 최소 1개 이상 있는가?
- [ ] **2단계** — 블로그 API와 무관한 독립 예제(Cat, User 등)가 있는가?
- [ ] **2단계** — 예제 코드가 직접 실행 가능한 수준인가?
- [ ] **3단계** — "블로그 API 적용" 또는 "실전" 섹션이 있는가?
- [ ] **3단계** — 이전 챕터에서 만든 코드와 연속성이 있는가?
- [ ] **3단계** — 동작 확인 방법(curl, 로그, 테스트)이 포함되어 있는가?
- [ ] **3단계** — 이미 만들어진 서비스/유틸리티를 재사용하고 있는가? (날짜 포맷 등 하드코딩 금지)
- [ ] **3단계** — 3단계 내용이 모두 `## 정리` **이전**에 위치하는가? (`## 정리` 뒤에 본문이 남아 있으면 구조 위반)
- [ ] **4단계** — `## 정리` 표가 있는가?
- [ ] **4단계** — `## 다음 챕터 예고`가 있는가? (챕터 17은 제외)
- [ ] **공통** — 단계 헤더가 H1(`#`)인가? (`##`로 쓰면 섹션으로 인식되어 구조가 무너짐)
- [ ] **공통** — TOC의 아이템 번호와 본문 섹션 번호가 일치하는가? (복붙 후 번호를 수정하지 않으면 불일치 발생)

---

## 2. 섹션 헤더 형식 (방식 A 단일 표준)

모든 챕터는 **방식 A** (명시적 단계 헤더)를 사용한다.

```markdown
---

# 1단계: 개념 학습

---

(개념 섹션들 — ## N. 형태로 번호 붙여도 됨)

---

# 2단계: 기본 예제

---

(독립 예제 섹션들)

---

# 3단계: 블로그 API 적용

---

(블로그 API 적용 섹션들)
```

각 단계 내부의 개별 섹션은 `## N. 섹션명` 형태로 번호를 붙여 세분화할 수 있다.

---

## 3. 파일 시작 구조 (고정)

```markdown
# 챕터 N - 주제명

> **이전 챕터 요약**: 챕터 N-1에서 ~를 만들었다. 이번 챕터에서는 ~를 학습한다.

## 목차
...
```

- 제목: `# 챕터 N - 주제명` 형식 (반드시 한글 "챕터")
- 이전 챕터 요약: 제목 바로 아래 blockquote (챕터 1 제외)

---

## 4. TOC 구조 (고정 4단계)

모든 챕터의 목차는 다음 형식을 따른다:

```markdown
## 목차

### 1단계: 개념 학습
1. [섹션명](#anchor)
...

### 2단계: 기본 예제
N. [섹션명](#anchor)
...

### 3단계: 블로그 API 적용
N+1. [섹션명](#anchor)
...

### 4단계: 정리
N+2. [정리](#정리)
N+3. [다음 챕터 예고](#다음-챕터-예고)  ← 챕터 17 제외
```

- 항목 번호는 챕터 전체에 걸쳐 연속된 숫자를 사용한다
- 챕터 17: `다음 챕터 예고` 항목 없음
- **앵커 정확도**: TOC 링크 `(#앵커)`는 본문 헤더에서 자동 생성된 앵커와 정확히 일치해야 한다. 섹션 번호를 바꾼 후 앵커를 수정하지 않으면 링크가 깨진다.
  - 예: 본문이 `## 14. PostgreSQL 전환 방법`이면 앵커는 `#14-postgresql-전환-방법`
  - 특수문자(`&`, `/`, `+`)는 앵커에서 제거됨에 주의

## 5. 파일 끝 구조 (고정)

```markdown
## 정리

| 개념 | 설명 |
|------|------|
| ... | ... |

## 다음 챕터 예고

챕터 N+1에서는 **주제**를 학습한다. 블로그 API에 ~기능이 추가된다.
```

- `## 정리` 표 → `## 다음 챕터 예고` 순서는 **절대 바뀌지 않는다**
- 챕터 17: `## 다음 챕터 예고` 없음
- **`## 정리` 이후에는 `## 다음 챕터 예고` 외 어떤 본문 내용도 두지 않는다** (심화 섹션, 보안 강화 등 추가 내용은 반드시 `## 정리` 앞에 배치)

---

## 6. 블록쿼트 스타일 (3종 고정)

```markdown
> **팁:** 추가 도움이 되는 정보
> **주의:** 실수하기 쉬운 부분
> **참고:** 배경 지식이나 보조 설명
```

**절대 금지 패턴:**
- `> **팁:**: ` — 콜론이 이중으로 들어가는 오류
- `> **팁::`** — 볼드 안에 콜론이 두 개
- `> **Note:**`, `> **Warning:**` — 영어 사용 금지
- `> **정리:**` — blockquote 형식의 정리 (반드시 `## 정리` 테이블로)

---

## 7. 코드 블록 규칙

```typescript
// src/경로/파일명.ts  ← 반드시 첫 줄에 파일 경로 주석
import { ... } from '@nestjs/common';
```

- 언어 태그: 항상 `typescript`
- 파일명 주석: **모든** 코드 블록 첫 줄에 포함 (예외: 개념 설명용 pseudocode)
- 터미널 명령은 `bash` 언어 태그 사용

---

## 8. 데코레이터 인라인 링크

코드 블록 **밖**에서 데코레이터 언급 시 반드시 링크 적용:

```markdown
[`@Controller()`](references/decorators.md#controllerprefix)
```

앵커 생성 규칙 (GitHub 마크다운):
- 소문자로 변환
- 공백 → `-`
- `&`, `/`, `(`, `)`, `,` 등 특수문자 제거
- 예: `## 2. 컨트롤러 & 라우팅 데코레이터` → `#2-컨트롤러-라우팅-데코레이터`

---

## 9. 챕터 연속성 규칙

- 각 챕터의 3단계(블로그 API 적용)는 **이전 챕터에서 만든 코드를 그대로 이어받는다**
- 이전 챕터에서 정의한 파일/클래스를 다시 처음부터 작성하지 않는다
- `// 이전 챕터에서 작성한 코드` 주석으로 연속성을 명시한다

## 10. 블로그 API 예제 코드 품질 규칙

### 하드코딩 최소화 — 기존 서비스/유틸리티 재사용

3단계(블로그 API 적용)의 예제 코드에서 이미 만들어진 서비스나 유틸리티가 있다면 반드시 재사용한다:

- **날짜 포맷**: `new Date().toISOString().split('T')[0]` 대신 `this.commonService.formatDate(new Date())`
- **슬러그 생성**: 인라인 문자열 처리 대신 `this.commonService.generateSlug(title)`
- **공통 로직**: 이미 서비스 레이어에 구현된 기능을 컨트롤러나 다른 서비스에서 중복 구현하지 않는다

```typescript
// ❌ 하드코딩 — 하지 말 것
createdAt: new Date().toISOString().split('T')[0],

// ✅ CommonService 재사용
createdAt: this.commonService.formatDate(new Date()),
```

### 의존성 주입 명시

재사용할 서비스를 사용하려면 생성자 주입을 명시한다:

```typescript
constructor(private readonly commonService: CommonService) {}
```

### 적용 기준

이 규칙은 **3단계: 블로그 API 적용** 코드에만 적용한다. 2단계(기본 예제)는 개념 설명용이므로 간결한 인라인 코드를 써도 된다.

## 빠른 참조: 모든 챕터 헤더 방식

모든 챕터(01~17, 신규 18~)는 **방식 A** (명시적 단계 헤더)를 사용한다.

```
# 1단계: 개념 학습
# 2단계: 기본 예제
# 3단계: 블로그 API 적용
```

단계 헤더는 H1(`#`)을 사용하며, 단계 내부의 세부 섹션은 H2(`##`)로 번호를 붙여 구분한다.
